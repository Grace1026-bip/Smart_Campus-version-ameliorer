SMART FACULTY
01.03 – Besoins non fonctionnels
Introduction
Les besoins non fonctionnels définissent les caractéristiques de qualité que Smart Faculty devra respecter afin d’offrir une plateforme performante, sécurisée, fiable et agréable à utiliser.
Ils ne décrivent pas les fonctionnalités du système, mais les exigences techniques et organisationnelles qui garantiront son bon fonctionnement.
1. Sécurité
La sécurité constitue une exigence essentielle du projet.
Le système devra :
protéger les comptes utilisateurs par une authentification sécurisée ;
chiffrer les mots de passe dans la base de données ;
contrôler les permissions selon le rôle de chaque utilisateur ;
empêcher les accès non autorisés ;
protéger les données académiques contre toute modification illégitime.
2. Performance
Le système devra offrir de bonnes performances afin d’assurer une utilisation fluide.
Il devra notamment :
afficher rapidement les informations ;
traiter efficacement les requêtes ;
gérer plusieurs utilisateurs simultanément ;
optimiser les échanges entre Flutter, FastAPI et MySQL.
3. Fiabilité
Smart Faculty devra garantir l’intégrité des informations académiques.
Le système devra :
éviter les pertes de données ;
assurer la cohérence des informations enregistrées ;
gérer correctement les erreurs ;
garantir la disponibilité des données.
4. Ergonomie
L’application devra être simple à utiliser.
L’interface devra :
être claire et intuitive ;
proposer une navigation simple ;
être adaptée aux différents utilisateurs ;
permettre un accès rapide aux fonctionnalités principales.
5. Maintenabilité
Le projet devra être facilement maintenable.
Le code devra :
être organisé en modules ;
être correctement documenté ;
être facilement modifiable ;
respecter une architecture claire.
6. Évolutivité
Smart Faculty devra pouvoir évoluer sans remettre en cause son architecture.
Le système devra permettre l’ajout futur de nouveaux modules tels que :
assistant académique intelligent ;
gestion des stages ;
gestion des TFC ;
Campus Analytics ;
orientation intelligente.
7. Compatibilité
Le système devra assurer une bonne compatibilité entre ses différents composants.
Il devra fonctionner avec :
Flutter pour le frontend ;
FastAPI pour le backend ;
MySQL pour la base de données ;
les principaux navigateurs web modernes.
8. Disponibilité
Le système devra rester accessible pendant les périodes d’utilisation académique.
Les opérations courantes devront pouvoir être réalisées sans interruption majeure.
9. Sauvegarde et traçabilité
Le système devra conserver un historique des opérations importantes.
Il devra permettre :
la sauvegarde des données ;
la journalisation des actions importantes ;
le suivi des modifications réalisées dans le système.
Conclusion
Les besoins non fonctionnels garantissent la qualité globale de Smart Faculty. Ils assurent que la plateforme ne sera pas seulement fonctionnelle, mais également sécurisée, performante, fiable et évolutive. Ces exigences serviront de référence lors du développement, des tests et de la maintenance du système.
