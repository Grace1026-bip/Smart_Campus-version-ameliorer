SMART FACULTY
01.02 – Besoins fonctionnels
Introduction
Les besoins fonctionnels décrivent l’ensemble des services que Smart Faculty doit fournir à ses utilisateurs. Ils définissent les fonctionnalités que le système devra réaliser afin de répondre aux besoins académiques d’une faculté universitaire.
Ces besoins constituent la base du développement de l’application et serviront de référence pour la conception, la programmation et les tests.
Besoin fonctionnel 1 : Authentification
Le système doit permettre aux utilisateurs de s’authentifier de manière sécurisée.
Le système doit permettre :
la connexion avec une adresse e-mail et un mot de passe ;
la déconnexion ;
la récupération du mot de passe ;
la gestion des sessions ;
la fonctionnalité « Se souvenir de moi » ;
la redirection automatique vers le tableau de bord correspondant au rôle de l’utilisateur.
Besoin fonctionnel 2 : Gestion des utilisateurs
Le système doit permettre :
la création des comptes ;
la validation des inscriptions ;
la gestion des rôles ;
la modification des informations personnelles ;
la désactivation d’un compte ;
la consultation des profils.
Les rôles pris en charge sont :
Administrateur ;
Doyen ;
Appariteur ;
Enseignant ;
Étudiant.
Besoin fonctionnel 3 : Gestion académique
Le système doit permettre :
la gestion des promotions ;
la gestion des cours ;
l’affectation des enseignants aux cours ;
la gestion des inscriptions des étudiants ;
la consultation des informations académiques.
Besoin fonctionnel 4 : Valve académique
Le système doit permettre aux enseignants :
de publier des annonces ;
de publier des documents ;
de publier les notes ;
de communiquer des informations importantes.
Le système doit permettre aux étudiants :
de consulter les publications ;
de télécharger les documents ;
de consulter les informations de chaque cours.
Besoin fonctionnel 5 : Gestion des notes
Le système doit permettre :
la création des évaluations ;
l’encodage des notes ;
la modification des notes avant publication ;
la publication des résultats ;
la consultation des notes par les étudiants.
Besoin fonctionnel 6 : Calcul académique
Le système doit calculer automatiquement :
la moyenne de chaque cours ;
la moyenne générale ;
les crédits validés ;
les crédits restants ;
les cours réussis ;
les cours échoués.
Besoin fonctionnel 7 : Réclamations académiques
Le système doit permettre :
la création d’une réclamation ;
le suivi de son évolution ;
la réponse des enseignants ou responsables ;
la clôture d’une réclamation.
Les statuts possibles sont :
En attente ;
En cours ;
Résolue ;
Rejetée.
Besoin fonctionnel 8 : Étudiants à risque
Le système doit analyser :
les notes ;
les absences ;
les retards ;
les moyennes.
Il doit identifier automatiquement les étudiants présentant un risque d’échec.
Les responsables concernés doivent être informés.
Besoin fonctionnel 9 : Tableau de bord
Le système doit fournir :
Pour l’étudiant :
ses notes ;
sa moyenne ;
ses crédits ;
ses alertes.
Pour l’enseignant :
ses cours ;
ses étudiants ;
les étudiants à risque ;
les publications.
Pour l’appariteur :
les promotions ;
les réclamations ;
les statistiques.
Pour le doyen :
les indicateurs académiques ;
les performances des promotions ;
les étudiants à risque ;
les statistiques globales.
Pour l’administrateur :
les utilisateurs ;
les rôles ;
les paramètres du système ;
les journaux d’activité.
Besoin fonctionnel 10 : Notifications
Le système doit envoyer automatiquement des notifications lors :
de la publication des notes ;
d’une nouvelle annonce ;
d’une nouvelle réclamation ;
d’une réponse à une réclamation ;
d’une alerte académique ;
d’une modification importante concernant un cours.
Synthèse des besoins fonctionnels
Les besoins fonctionnels identifiés garantissent que Smart Faculty répondra aux attentes des différents utilisateurs tout en assurant une gestion académique moderne, centralisée et efficace.
Chaque besoin décrit dans ce document sera développé sous forme de modules au cours de la phase de réalisation.
Conclusion
Les besoins fonctionnels constituent la référence principale pour le développement de Smart Faculty. Ils décrivent précisément les services que le système devra offrir et serviront de base pour la conception technique, le développement des modules et la validation finale du projet.
